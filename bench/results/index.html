Invariant tests

Pass/fail assertions modeled on the deep-dive's "synthetic users with known preference profiles" pattern. They gate the build — a real failure here means the recommender is doing the wrong thing on at least one well-defined user archetype.

  • ✅ PASS · category-locked — score 0.9992, threshold 0.4000. mean fraction of top-10 in the user's locked category
  • ❌ FAIL · diversity-floor — score 1.0000, threshold 0.9500. worst single-category share of any user's top-10; threshold=0.95
  • ✅ PASS · beats-random — score 0.0105, threshold 0.0015. two-tower ndcg@10=0.0120 vs random ndcg@10=0.0015; require >2x

Two-tower centerpiece

PyTorch user-tower + org-tower with in-batch sampled-softmax + popularity-weighted negative sampling. L2-normalized outputs, FAISS exact inner-product for top-K retrieval. The two-tower-content-init ablation initializes the org tower from sentence-transformers/all-MiniLM-L6-v2 embeddings before fine-tuning.

Reproduce locally

git clone https://github.com/d-malhotra2020/donation-platform

cd donation-platform && make bench

Fixed seeds, pinned dependencies (bench/requirements.txt), CPU-only by contract. Same git SHA → same metrics.json modulo runtime.

What this is, and what it isn't

  • The org corpus is a sanitized snapshot of ProPublica's Nonprofit Explorer from 2026-05-24. ~3K orgs sampled per-category from the full 5K snapshot for the headline run. CSV + fetch script live in the repo at bench/data/.
  • The users and donations are synthetic, generated from the seeded process in bench/data/synthetic_*.py. Not real people, not real giving behavior.
  • All eval metrics are computed on the synthetic test split. They measure model quality on this synthetic giving pattern. They do not represent performance on any real production donation platform.
  • This is Slice 1 of a four-slice rebuild. Slices 2–4 (FastAPI gateway, Redis embedding cache, web demo surface) are out of scope here and will land in follow-up cycles.
